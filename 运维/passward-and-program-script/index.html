Accessibility settings

1 January 0001·471 words·3 mins· loading · loading ·

最近在写GUI工具，希望动态注册 systemd 的服务，还有一些如编辑/var/libn德国需要 sudo 权限的功能。于是研究下如何通传sudo权限。

script
#

脚本拿sudo权限相对简单。要么作为用户直接使用sudo运行脚本，要么在脚本里直接写 “sudo command”。

Bash

if [ "$EUID" -ne 0 ]; then
  echo "请使用 sudo 运行此脚本"
  exit 1
fi

或者直接在sh里写，会像普通sudo一样要求密码

Bash

# 执行需要权限的操作 
sudo dnf update

但是这样写会有问题，如下面这个脚本演示

Bash

#!/bin/bash

# 获取脚本的绝对路径，确保嵌套调用时能准确找到它本身
SCRIPT_PATH=$(readlink -f "$0")
LEVEL=${1:-0}

# 定义一个打印核心环境变量和路径的函数
print_env() {
    echo -e "\n=== 当前层级: $1 ==="
    echo "实际执行身份 (whoami) : $(whoami)"
    echo "有效用户ID (EUID)     : $EUID"
    echo "环境变量 \$USER        : $USER"
    echo "环境变量 \$SUDO_USER   : ${SUDO_USER:-<未设置>}"
    echo "家目录 \$HOME          : $HOME"
    echo "当前工作目录 \$PWD     : $PWD"
    echo "-----------------------"
    echo "⚠️ 自定义变量 \$MY_CUSTOM_VAR: ${MY_CUSTOM_VAR:-<被 sudo 清理/丢失>}"
    echo "📊 环境变量总数量 (env count): $(env | wc -l) 个"
    echo "======================="
}

case "$LEVEL" in
    0)
        echo "🔵 [测试开始] 初始状态：普通用户执行"
        # 在 Level 0 导出一个自定义环境变量，看它能不能活到后面
        export MY_CUSTOM_VAR="我是在_Level_0_定义的变量"
        print_env "Level 0 (普通用户)"

        echo "🟢 [进入第一层] 触发 Level 1：普通 sudo"
        sudo "$SCRIPT_PATH" 1
        ;;
    1)
        print_env "Level 1 (普通 sudo)"

        echo "🟡 [进入第二层] 触发 Level 2A：在 sudo 内部再次执行普通 sudo"
        sudo "$SCRIPT_PATH" 2a

        echo "🟠 [进入第二层] 触发 Level 2B：在 sudo 内部执行 sudo -i (模拟登录环境)"
        sudo -i "$SCRIPT_PATH" 2b
        ;;
    2a)
        print_env "Level 2A (sudo 嵌套 sudo)"
        ;;
    2b)
        print_env "Level 2B (sudo 嵌套 sudo -i)"
        ;;
    *)
        echo "未知层级"
        ;;
esac

二进制程序与 GUI 工具如何获取 Root 权限
#

对于使用 Go、Rust、C++ 或 Python (打包为二进制) 编写的 GUI 工具，直接在代码里调用 sudo command 体验很差，因为终端的密码提示无法在图形界面中展示（除非强制弹出一个终端窗口）。

有以下几种主流且优雅的解决方案：

1. 使用 Polkit (pkexec) —— 桌面 GUI 的首选
#

Polkit (PolicyKit) 是现代 Linux (包括 Rocky Linux, Ubuntu 等) 桌面环境处理权限提升的标准组件。

当你用 pkexec 代替 sudo 执行命令时，如果处于图形界面，系统会自动弹出一个原生的 GUI 密码输入框（由 KDE/GNOME 提供）。

代码层面的调用：

在代码中执行外部命令时，将 sudo systemctl start svc 替换为：

Bash

pkexec systemctl start svc

优点： 完美融入 GUI 体验，无需自己写密码输入框。

缺点： 每次调用 pkexec 都会弹窗，如果你的程序需要频繁执行特权操作，用户会被烦死。

2. 前后端分离架构 (Daemon + Frontend) —— 最符合工程规范
#

如果 GUI 需要长期、高频地执行 Root 操作，最佳实践是不要让 GUI 进程本身获取 Root 权限。

  • 后端 (Daemon)： 编写一个没有界面的服务程序，通过 Systemd 注册为以 root 身份运行的后台服务。

  • 前端 (GUI)： 以普通用户身份启动。

  • 通信： 前端通过 Unix Domain Sockets、本地 D-Bus 或 gRPC 向后端发送指令（如“请帮我修改 /var/lib/xxx”），后端执行后返回结果。

优点： 安全性最高（GUI 崩溃不会导致系统级漏洞），完全没有频繁弹窗的问题，这也是 Docker Desktop、各类 VPN 客户端的标准做法。

3. SUID (Set-owner-User-ID) —— 简单粗暴的二进制提权
#

如果是编译型的二进制文件（C/C++, Go, Rust，不适用于 Shell/Python 脚本），可以通过设置 SUID 标志位，让普通用户执行它时，自动继承文件所有者 (root) 的权限。

Bash

sudo chown root:root /path/to/your/binary
sudo chmod u+s /path/to/your/binary

注意： 这种做法极不安全，这意味着任何用户运行该工具都会直接获得 Root 权限。通常只用于非常底层且严格限制输入的小型工具（如系统的 ping 或 passwd 命令），不建议用于复杂的 GUI 工具。

如何给运维脚本/程序配备 sudo 权限 (免密执行)
#

在自动化运维（如 CI/CD、Ansible、Cron 定时任务或监控脚本）中，由于没有人工干预，脚本不能卡在等待密码的步骤。在 Rocky Linux 及所有现代主流发行版中，通用的解决办法是配置 sudoers。

1. 配置 /etc/sudoers.d/
#

不要直接编辑 /etc/sudoers 文件，而是使用 visudo 在 /etc/sudoers.d/ 目录下创建一个新文件。这种方式易于管理，且卸载脚本时直接删除对应文件即可。

操作步骤：

使用 root 或 sudo 执行：

Bash

sudo visudo -f /etc/sudoers.d/my_ops_script

写入以下内容（假设执行脚本的用户是 appuser）：

Plaintext

# 允许 appuser 免密执行特定的脚本
appuser ALL=(ALL) NOPASSWD: /opt/scripts/my_ops_script.sh

# 允许 appuser 免密执行 systemctl 重启特定服务
appuser ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart my-service.service

在脚本中的调用：

配置完成后，脚本中仍然需要显式写 sudo，但系统不会再要求输入密码：

Bash

#!/bin/bash
sudo /usr/bin/systemctl restart my-service.service

注意：在 sudoers 中必须写命令的绝对路径（如 /usr/bin/systemctl），否则会有安全风险且可能被策略拒绝。

2. 针对 Systemd 服务的特殊方案：自定义 Polkit 规则
#

如果你的运维脚本仅仅是为了管理 Systemd 服务（启动、停止、重启），除了 sudoers，还可以通过配置 Polkit 规则来实现免密，这在 Rocky Linux (RHEL 系) 中非常正规。

在 /etc/polkit-1/rules.d/ 下创建规则文件，例如 10-manage-myservice.rules：

JavaScript

polkit.addRule(function(action, subject) {
    if (action.id == "org.freedesktop.systemd1.manage-units" &&
        action.lookup("unit") == "myservice.service" &&
        subject.user == "appuser") {
        return polkit.Result.YES;
    }
});

效果： 用户 appuser 可以直接运行 systemctl restart myservice，甚至连 sudo 都不用加，也不会提示输入密码。这比开放 sudo 权限的粒度更细，也更安全。
